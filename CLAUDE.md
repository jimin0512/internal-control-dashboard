# CLAUDE.md

이 문서는 내부회계관리제도 분석 위키의 운영 지침이다. 이 위키에서 작업할 때는 아래 규칙을 따른다.

## 위키 목적

이 위키는 내부회계관리제도의 프로세스, 통제, 설계평가, 운영평가, 미비점 및 개선조치 데이터를 분석하고 지식을 축적하기 위한 개인 위키다.

단순한 데이터 조회에 그치지 않고 다음 업무를 지원하는 것을 최종 목적으로 한다.

- 내부회계 설계평가 및 운영평가 결과 분석
- 프로세스별·부서별·통제별 현황 분석
- Key 통제 및 주요 미비점 관리
- 미비점 개선조치 진행현황 관리
- 반복 미비점 및 지연 개선조치 식별
- 내부회계 운영보고서 자동 작성
- 경영진 보고자료 및 외부감사 대응자료 작성 지원

## 작업 흐름

이 위키는 다음 순서로 지식을 축적한다.

**raw → schema → question → analysis → insight → report**

각 단계의 역할은 명확히 분리되며, 뒷 단계는 앞 단계의 결과만 인용하고 스스로 raw를 다시 계산하지 않는다.

| 단계 | 폴더 | 역할 | 계산 여부 |
|---|---|---|---|
| raw | `raw/` | 원본 CSV | 수정 금지 |
| schema | `02_data/` | raw의 구조·의미 정리 | 계산하지 않음 |
| question | `01_questions/` | 질문, 목적, 필요한 데이터, 계산 방법만 정의 | 계산하지 않음 |
| analysis | `03_analysis/` | raw CSV를 직접 계산, 집계표·계산 결과 작성 | 직접 계산함 |
| insight | `04_insights/` | analysis 결과만 근거로 해석·시사점 작성 | 계산하지 않음 (새 수치 생성 금지) |
| report | 위키 루트 (`report-topic-date.md`) | question·analysis·insight를 종합해 보고서 형태로 재구성 | 계산하지 않음 (analysis·insight 결과만 인용) |

각 단계별 세부 작성 규칙은 아래 "스키마 노트 작성 규칙", "질문 노트 작성 규칙", "분석 노트 작성 규칙", "인사이트 노트 작성 규칙", "자동보고서 작성 규칙"을 따른다.

## 폴더 구조

```
raw/            원본 CSV 파일 (수정 금지)
00_inbox/       미분류 요약 노트, 검토 요청 메모
01_questions/   질문 노트
02_data/        스키마 노트
03_analysis/    분석 노트
04_insights/    인사이트 노트
05_log/         날짜별 작업 로그
```

루트에는 다음 파일이 있다.

- `INDEX.md`: 위키 전체의 허브 노트
- `CLAUDE.md`: 이 문서. 위키 운영 지침
- `report-topic-date.md`: 보고서 노트. 별도 폴더 없이 위키 루트에 저장한다.
- `.claude/commands/wiki-sync.md`: 신규 원본을 증분 처리하는 커스텀 명령 (`/wiki-sync`)

## 파일 및 노트 이름 규칙

- CSV 원본 파일명은 영문 소문자와 언더바를 사용한다.
- 스키마 노트 파일명: `schema-원본파일명.md` (원본파일명은 raw CSV 파일명을 그대로 따르므로 영문을 유지한다. 예: `schema-ic_control_master.md`)
- 질문 노트 파일명: `q-001-토픽.md` (예: `q-001-프로세스별통제현황.md`)
- 분석 노트 파일명: `analysis-001-토픽.md` (근거가 된 질문 노트와 동일한 번호·토픽을 사용한다. 예: `analysis-001-프로세스별통제현황.md`)
- 인사이트 노트 파일명: `insight-001-토픽.md` (근거가 된 질문·분석 노트와 동일한 번호·토픽을 사용한다. 예: `insight-001-프로세스별통제현황.md`)
- 보고서 노트 파일명: `report-토픽-날짜.md`
- 번호 접두어(`q-001`, `analysis-001`, `insight-001`)와 하이픈, 확장자는 영문/숫자를 유지하고, 토픽 부분은 한글로 작성한다.
- 노트 제목과 본문은 한글로 작성한다.
- CSV의 한글 컬럼명은 원본 그대로 유지하며 임의로 변경하지 않는다.
- YAML key는 영문으로 작성한다.

## 데이터 연결 구조

데이터는 다음 순서로 연결한다.

- `ic_process_master.프로세스ID` → `ic_control_master.프로세스ID`
- `ic_control_master.통제ID` → `ic_design_assessment.통제ID` → `ic_operating_assessment.통제ID` → `ic_deficiency.통제ID` → `ic_remediation.통제ID`
- `ic_operating_assessment.운영평가ID` → `ic_deficiency.운영평가ID`
- `ic_deficiency.미비점ID` → `ic_remediation.미비점ID`

분석 시 **프로세스 → 통제 → 설계평가·운영평가 → 미비점 → 개선조치** 순서로 추적한다.

## 원본 자료 처리 규칙

- `raw/` 폴더의 원본 파일은 절대 수정하지 않는다.
- `raw/`에 새 파일이 추가되면 raw 파일 목록과 `02_data/`, `00_inbox/`의 노트 목록을 파일명 기준으로 먼저 대조한다.
- 이 목록 대조 단계에서는 원본 내용을 읽지 않는다.
- 이미 대응 노트가 있는 파일은 다시 읽지 않는다.
- 대응 노트가 없는 신규 파일만 읽는다.
- 표 형태 데이터는 `02_data/`에 스키마 노트를 생성한다.
- 표가 아닌 자유 형식 자료는 `00_inbox/`에 요약 노트를 생성한다.
- 내용이나 컬럼의 의미가 불분명하면 추측하지 말고 `00_inbox/`에 검토 요청 메모를 생성한다.
- 생성한 노트에는 반드시 `source: [[raw/원본파일명]]`을 기록한다.
- 공통 연결키가 있는 데이터는 관련 스키마 노트를 Obsidian 위키링크로 연결한다.
- 한글 컬럼명은 원본 그대로 사용한다.
- 원본 데이터의 값을 임의로 수정, 보완 또는 추정하지 않는다.

## 스키마 노트 작성 규칙

각 스키마 노트에는 다음 내용을 포함한다.

- 데이터의 한 줄 정의
- 전체 행 수
- 기본키와 연결키
- 필드 목록
- 각 필드의 의미
- 데이터 간 연결 관계
- 분석 시 주의사항
- 자동보고서에서 활용 가능한 항목
- 관련 질문 후보

모든 스키마 노트 상단에는 다음 YAML frontmatter를 작성한다.

```yaml
---
date:
category: data
table:
rows:
key:
source:
tags:
---
```

## 질문 노트 작성 규칙

질문은 다음 4단계 성숙도 모델을 사용한다.

1. 기술적 질문: 현재 어떤 현황인가
2. 진단적 질문: 왜 이런 결과가 발생했는가
3. 예측적 질문: 향후 어떤 위험이나 지연이 예상되는가
4. 처방적 질문: 어떤 조치와 우선순위가 필요한가

하나의 질문 노트는 이 4단계를 모두 아우르도록 설계한다(하나의 질문이 기술적 질문에서 출발해 진단적·예측적·처방적 단계까지 이어진다). frontmatter의 `maturity`는 `[기술적, 진단적, 예측적, 처방적]`처럼 배열로 기재한다. 질문은 실제 raw 데이터로 계산 가능하거나, analysis 노트로 연결해 분석 가능한 내용만 작성한다.

질문 노트는 다음 역할만 담당한다. 다른 단계의 산출물을 질문 노트에 적지 않는다.

- 질문 노트에는 실제 계산 결과·집계값·해석·시사점을 작성하지 않는다.
- 질문 노트는 질문, 질문 목적, 필요한 데이터(관련 스키마 노트 위키링크), 계산 또는 분석 방법까지만 작성한다.
- 실제 계산은 반드시 `03_analysis`에 별도 분석 노트로 작성한다.
- 해석과 시사점은 반드시 `04_insights`에 작성한다.
- 전체 작업 흐름은 위 "작업 흐름" 절의 raw → schema → question → analysis → insight → report 순서를 따른다.

모든 질문 노트 상단에는 다음 YAML frontmatter를 작성한다.

```yaml
---
date:
category: question
maturity:
priority:
data:
tags:
---
```

본문은 다음 순서로 작성한다.

- 질문
- 질문 목적
- 필요한 데이터
- 계산 또는 분석 방법 (실제 계산값은 작성하지 않는다)
- 관련 분석 (해당 분석 노트가 없으면 "추후 03_analysis에 생성"이라고 적는다)
- 관련 인사이트 (해당 인사이트 노트가 없으면 "추후 04_insights에 생성"이라고 적는다)

## 분석 노트 작성 규칙

분석 노트는 raw CSV를 직접 읽어 계산·조인하는 유일한 단계다. 질문 노트와 인사이트 노트는 계산하지 않으므로, 실제 수치가 필요한 모든 작업은 분석 노트에서 수행한다.

- raw CSV를 직접 읽어 계산한다. schema 노트와 question 노트를 참고하되, 계산 자체는 raw 데이터를 기준으로 한다.
- 집계표, 비율, 비교 결과를 작성한다. 모든 비율은 분자와 분모를 함께 표시한다(예: `174 / 180 (96.7%)`).
- 어떤 컬럼을 어떤 기준으로 집계·연결했는지 분석 방법을 명확히 남긴다.
- 데이터에서 직접 확인된 사실만 "발견사항"에 작성하고, 원인 해석이나 시사점은 작성하지 않는다. 해석과 시사점은 `04_insights`에 작성한다.
- 관련 질문 노트와 사용한 스키마 노트를 Obsidian 위키링크로 연결한다.
- 실제 raw 데이터로 계산 가능한 값만 사용하고, 추측하지 않는다.

모든 분석 노트 상단에는 다음 YAML frontmatter를 작성한다.

```yaml
---
date:
category: analysis
question:
source_data:
tags:
---
```

본문은 다음 순서로 작성한다.

- 관련 질문 (질문 노트 위키링크)
- 사용 데이터 (관련 스키마 노트 위키링크)
- 분석 방법 (어떤 컬럼을 어떤 기준으로 집계했는지)
- 계산 결과 (실제 계산 결과. 표 형태 포함, 비율은 분자/분모 형식)
- 발견사항 (데이터에서 확인된 사실만 작성. 해석하지 않는다)
- 다음 단계 (관련 인사이트 노트 위키링크. 해당 노트가 없으면 "추후 04_insights에 생성"이라고 적는다)

## 인사이트 노트 작성 규칙

인사이트 노트는 다음 원칙을 따른다.

- Analysis 노트의 결과만 근거로 사용한다.
- raw CSV를 직접 계산하지 않는다.
- 새로운 수치나 집계를 만들지 않는다. 필요하면 `03_analysis`에 분석 노트를 먼저 만들거나 보완한다.
- 근거(사실)와 해석(의견)을 구분한다.

인사이트는 반드시 **근거 → 해석 → 시사점** 순서로 작성한다.

- 근거: 관련 분석 노트에서 이미 확인된 사실을 인용·요약한다. 새로운 계산은 하지 않는다.
- 해석: 근거가 의미하는 내용을 설명한다.
- 시사점: 내부회계 운영 또는 개선조치에 필요한 행동을 작성한다.

추측과 사실을 구분한다.

- 분석 노트에서 직접 계산으로 확인된 내용은 사실로 표현한다.
- 원인이나 향후 영향에 대한 해석은 "~로 보인다", "~일 가능성이 있다"라고 표현한다.

모든 인사이트 노트 상단에는 다음 YAML frontmatter를 작성한다.

```yaml
---
date:
category: insight
confidence:
source_question:
source_analysis:
tags:
---
```

본문은 다음 순서로 작성한다.

- 관련 질문 (질문 노트 위키링크)
- 관련 분석 (분석 노트 위키링크)
- 근거
- 해석
- 시사점
- 우선순위 (높음 / 중간 / 낮음)

confidence 기준은 다음과 같다.

- 높음: 분석 노트의 raw 데이터 직접 계산 결과만으로 뒷받침되는 해석
- 중간: 분석 노트의 계산 결과에 합리적 해석을 더해 작성
- 낮음: 근거가 되는 분석 노트의 표본이 부족하거나 가설 중심으로 작성

## 분석 및 KPI 기준

다음 지표는 `03_analysis`의 분석 노트에서 raw 데이터로 직접 계산하는 것을 우선한다. 질문 노트는 이 지표들을 계산 대상으로 지정하고, 인사이트 노트와 자동보고서는 분석 노트에서 이미 계산된 값을 인용한다.

### 1. 프로세스 및 통제 현황
- 프로세스별 전체 통제 수
- 프로세스별 Key 통제 수 및 비율
- 통제유형, 통제방법, 수행주기별 분포
- 통제책임부서와 통제수행부서별 통제 수
- 신규통제 및 사용중 통제 현황

### 2. 설계평가
- 전체 설계 적정률
- 프로세스별 설계 적정률
- Key 통제 설계 적정률
- 설계 미흡 통제 및 개선필요사유
- RCM 미반영 통제 현황

### 3. 운영평가
- 평가차수별 운영 적정률
- 프로세스별 운영 적정률
- Key 통제 운영 적정률
- 표본수 대비 예외건수 및 예외율
- 증빙 미완료 현황
- 기한 내 미제출 현황
- 검토 미완료 현황
- 평가차수별 개선 또는 악화 추이

### 4. 미비점
- 프로세스별 미비점 수
- 중요도별 미비점 수
- 미비점유형별 현황
- 반복 미비점 현황
- 근본원인별 미비점 현황
- 개선담당부서별 미비점 수
- 미종결 미비점 현황

### 5. 개선조치
- 전체 개선 완료율
- 진행상태별 개선조치 수
- 평균 진행률
- 목표완료일 경과 여부
- 기한 경과 미완료 개선조치
- 검증 필요 미완료 건
- 검증결과가 미흡하거나 미입력된 건
- 담당부서 및 담당자별 개선조치 현황

비율 계산 시 반드시 분자와 분모를 함께 표시한다.

예:
- 운영 적정률 92.5% (500건 / 540건)
- Key 통제 비율 45.0% (81개 / 180개)

표본 수가 지나치게 적거나 결측값이 있는 경우 분석의 한계를 함께 작성한다.

## 자동보고서 작성 규칙

사용자가 운영보고서 또는 평가결과 보고서를 요청하면 다음 구조를 기본으로 한다.

1. 평가 개요
2. 프로세스 및 통제 현황
3. 설계평가 결과
4. 운영평가 결과
5. 주요 미비점
6. 개선조치 진행현황
7. 핵심 위험 및 관리 필요사항
8. 종합 의견
9. 향후 일정 및 요청사항

보고서 노트는 별도 폴더 없이 위키 루트에 `report-topic-date.md` 형식으로 저장한다.

보고서에는 다음 원칙을 적용한다.

- raw 데이터를 직접 계산하지 않는다. `03_analysis`에서 이미 계산된 값과 `04_insights`의 해석·시사점만 확정 수치·의견으로 인용한다.
- 모든 비율에 분자와 분모를 표시한다.
- 전기 또는 이전 차수 비교가 가능한 경우 추이를 함께 표시한다.
- 중요도 상, 반복 미비점, 기한 경과 개선조치를 우선 표시한다.
- 경영진이 판단해야 할 사항과 실무자가 조치해야 할 사항을 구분한다.
- 사실(analysis 노트 근거), 해석·권고사항(insight 노트 근거)을 구분한다.
- 표와 요약문을 함께 작성한다.
- 근거 데이터와 관련 스키마·질문·분석·인사이트 노트를 위키링크로 연결한다.

## 작업 로그 규칙

- 세션 종료 시 `05_log/오늘날짜.md`에 작업 내용을 기록한다.
- 동일 날짜의 로그가 있으면 새 파일을 만들지 말고 기존 파일에 이어서 기록한다.
- 로그에는 다음을 포함한다.
  - 처리한 raw 파일
  - 생성한 노트
  - 수정한 노트
  - 수행한 계산 또는 분석
  - 확인이 필요한 사항
- 새 노트를 생성할 때 `INDEX.md`의 해당 섹션에 링크를 추가한다.

## INDEX.md 관리 규칙

`INDEX.md`는 다음 섹션을 유지한다.

```
## 데이터
## 검토 필요
## 질문
## 분석
## 인사이트
## 최근 로그
```

새 노트가 생성될 때마다 해당 섹션에 위키링크를 추가한다.
중복 링크는 만들지 않는다.
