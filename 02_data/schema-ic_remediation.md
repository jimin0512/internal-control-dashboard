# ic_remediation 스키마 노트

미비점별 개선조치 및 검증 현황을 담은 데이터다.

## 전체 행 수

41건 (미비점 41건과 1:1 관계로 보인다)

## 기본키와 연결키

- 기본키: 개선조치ID (예: A0001)
- 연결키: 미비점ID → [[schema-ic_deficiency]]
- 연결키: 통제ID → [[schema-ic_control_master]]

## 필드 목록 및 의미

| 필드 | 의미 |
|---|---|
| 개선조치ID | 개선조치 고유 식별자 |
| 미비점ID | 대상 미비점 ID |
| 통제ID | 대상 통제 ID |
| 개선조치내용 | 개선조치 내용 서술 |
| 담당부서 | 개선조치 담당 부서 |
| 담당자 | 개선조치 담당자 |
| 목표완료일 | 개선조치 목표 완료일 |
| 실제완료일 | 실제 완료일. 미완료 건은 공란 |
| 진행률 | 진행률(%) |
| 진행상태 | 완료 28건, 진행중 8건, 지연 5건 |
| 검증필요여부 | 검증 필요 여부. Y 26건, N 15건 |
| 검증결과 | 적정 27건, 추가보완 1건, 공란(미입력) 13건 |
| 비고 | 비고 |

## 데이터 간 연결 관계

- `ic_deficiency.미비점ID` → `ic_remediation.미비점ID`
- 통제ID는 미비점을 거치지 않고도 [[schema-ic_control_master]]와 직접 연결된다.

## 분석 시 주의사항

- 검증결과 공란 13건은 검증필요여부 Y이면서 아직 검증결과가 입력되지 않은 건일 가능성이 높으므로, 반드시 검증필요여부와 함께 교차 확인한 뒤 "검증 미완료"로 분류한다.
- 진행상태가 "완료"인데 실제완료일이 공란인 경우가 있는지 등 값 정합성은 raw 그대로 다루고 임의로 보정하지 않는다.
- 목표완료일 대비 현재상태(지연 5건)를 볼 때, 기준일(오늘 날짜) 대비 목표완료일 경과 여부를 별도로 계산해야 정확한 "기한 경과" 건수를 산출할 수 있다.

## 자동보고서에서 활용 가능한 항목

- 전체 개선 완료율(완료 28건/41건), 진행상태별 개선조치 수, 평균 진행률, 목표완료일 경과 미완료 건, 검증 필요 미완료 건, 담당부서·담당자별 개선조치 현황 — 이 파일 단독 계산 가능
- 미비점 중요도·유형별 개선 완료율: [[schema-ic_deficiency]] 연결 필요 (연결키: 미비점ID)
- 통제 Key여부별 개선 완료율: [[schema-ic_control_master]] 연결 필요 (연결키: 통제ID)

## KPI 연결 구조

```
반복 미비점 (ic_deficiency.csv, 반복미비점여부 컬럼)
→ 개선조치 완료율 (본 파일, 진행상태 컬럼)
→ 기한 경과 미완료 건 (본 파일, 목표완료일 vs 기준일 + 진행상태)
```

## 관련 질문 후보

- (기술적) 전체 개선 완료율과 평균 진행률은 얼마인가?
- (기술적) 검증이 필요하지만 검증결과가 미입력된 건은 몇 건인가?
- (진단적) 지연 상태인 개선조치들은 어떤 부서·근본원인에 집중되어 있는가?
- (진단적) 검증필요여부 Y인 건 중 검증결과가 "적정"이 아닌 건(추가보완·미입력)은 어떤 특징이 있는가?
- (예측적) 현재 진행중/지연 상태인 개선조치 중 목표기한을 넘길 가능성이 높은 건은 무엇인가?
- (예측적) 반복 미비점([[schema-ic_deficiency]])에 대한 개선조치가 다시 지연될 위험이 있는가?
- (처방적) 검증 미완료 및 기한 경과 개선조치에 대해 어떤 우선순위로 후속 조치를 해야 하는가?
- (처방적) 특정 담당자·담당부서에 지연 건이 몰려 있다면 업무 재분배가 필요한가?
