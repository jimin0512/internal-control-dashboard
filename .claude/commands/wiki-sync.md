raw/ 폴더와 02_data/, 00_inbox/ 사이의 차이를 확인하고, 새로 들어온 원본 자료를 정리해서 위키에 반영한다. 아래 순서를 반드시 지킨다.

1. **파일 목록 대조 (내용 읽지 않음)**
   - `raw/` 안의 모든 파일 목록을 가져온다.
   - `02_data/`와 `00_inbox/`에 있는 노트들의 파일명(또는 frontmatter의 source 파일명)을 가져온다.
   - 파일명 기준으로 대조해서, 아직 대응하는 노트가 없는 `raw/` 파일만 골라낸다.
   - 이 단계에서는 파일 내용을 열어보지 않는다.

2. **새 파일만 읽기**
   - 1단계에서 골라낸, 대응 노트가 없는 새 파일만 실제로 읽는다.
   - 대응 노트가 이미 있는 파일은 읽지 않고 건너뛴다.

3. **형식에 따라 노트 생성**
   - 표 형태 데이터(csv, tsv, xlsx, 표 형태 md/json 등)면 `02_data/`에 스키마 노트를 만든다. 아래 "스키마 노트 필수 구성"의 10개 항목을 모두 포함해야 한다.
   - 표가 아닌 자유 형식 자료(메모, 문서, 텍스트 등)면 `00_inbox/`에 요약 노트를 만든다. 핵심 내용을 간단히 요약하고 원본 파일 경로를 남긴다.

### 스키마 노트 필수 구성

`02_data/`에 스키마 노트를 생성할 때는 다음 10개 항목을 반드시 이 순서로 포함한다.

1. 한 줄 정의
2. 전체 행 수
3. 기본키와 연결키
4. 필드 목록 및 각 필드의 의미
5. 데이터 간 연결 관계
6. 분석 시 주의사항
7. 자동보고서에서 활용 가능한 항목
8. 보고서에서 사용될 섹션
9. KPI 연결 구조
10. 관련 질문 후보

각 항목의 작성 방법은 다음과 같다.

**7. 자동보고서에서 활용 가능한 항목**

이 데이터에서 직접 산출하거나 다른 데이터와 연결해 분석할 수 있는 지표를 구체적으로 적는다.

예: 프로세스별 통제 수, Key 통제 수 및 비율, 설계 적정률, 운영 적정률, 미비점 수와 발생률, 반복 미비점 수, 개선조치 완료율, 기한 경과 개선조치 수

이 파일 하나만으로는 계산할 수 없고 다른 스키마 노트와 연결해야 계산되는 지표는 반드시 어느 노트와 연결해야 하는지 위키링크로 표시한다.

예:
- 운영 적정률 계산: [[schema-ic_operating_assessment]]
- 미비점 수 계산: [[schema-ic_deficiency]]
- 개선 완료율 계산: [[schema-ic_remediation]]

**8. 보고서에서 사용될 섹션**

이 데이터가 내부회계 운영보고서의 어느 섹션에서 사용되는지 명시한다. 보고서 기본 섹션은 다음과 같다.

1. 평가 개요
2. 프로세스 및 통제 현황
3. 설계평가 결과
4. 운영평가 결과
5. 주요 미비점
6. 개선조치 진행현황
7. 핵심 위험 및 관리 필요사항
8. 종합 의견
9. 향후 일정 및 요청사항

예:
- ic_control_master: 2. 프로세스 및 통제 현황
- ic_design_assessment: 3. 설계평가 결과
- ic_operating_assessment: 4. 운영평가 결과
- ic_deficiency: 5. 주요 미비점, 7. 핵심 위험 및 관리 필요사항
- ic_remediation: 6. 개선조치 진행현황, 9. 향후 일정 및 요청사항

**9. KPI 연결 구조**

다음 KPI 흐름 중 이 데이터가 해당되는 연결관계를 표시한다.

```
프로세스별 통제 수
→ Key 통제 비율
→ 설계 적정률
→ 운영 적정률
→ 미비점 수 및 발생률
→ 반복 미비점
→ 개선조치 완료율
→ 기한 경과 미완료 건
```

각 KPI 항목마다 계산에 필요한 원본 파일명과 연결키(예: 통제ID, 운영평가ID, 미비점ID)를 함께 적는다.

**10. 관련 질문 후보**

최소 8개의 질문 후보를 작성한다. 질문은 다음 4단계로 구분하고 각 단계별 최소 2개씩 작성한다.

- 기술적: 현재 현황은 어떠한가
- 진단적: 왜 이런 결과가 발생했는가
- 예측적: 향후 어떤 위험이나 지연이 예상되는가
- 처방적: 어떤 조치와 우선순위가 필요한가

질문은 반드시 현재 raw 데이터로 계산하거나 다른 스키마 노트와 연결해 분석할 수 있는 내용으로만 작성한다. 근거가 없는 원인 추정성 질문은 만들지 않는다.

**파일 및 표기 규칙**

- 스키마 파일명은 영문 소문자로 작성한다(`schema-원본파일명.md`).
- 노트 제목과 본문은 한글로 작성한다.
- 한글 컬럼명은 원본 그대로 사용한다.
- Obsidian 위키링크는 실제 스키마 파일명과 정확히 일치시킨다.
- source는 반드시 `[[raw/원본파일명]]` 형식으로 기록한다.

4. **판단이 어려운 경우**
   - 내용이 불분명하거나 표인지 자유 형식인지, 또는 의미를 추측하기 어려운 경우 절대 추측해서 넘어가지 않는다.
   - 이런 파일은 `00_inbox/`에 "검토 요청" 메모를 만든다. 무엇이 불분명한지, 어떤 판단이 필요한지를 명확히 적는다.

5. **위키링크 연결**
   - 새로 만든 `02_data/` 노트들 중 공통 컬럼(같은 이름 또는 명백히 같은 의미의 컬럼)을 가진 것들이 있으면, 서로 `[[파일명]]` 형식의 위키링크로 연결한다.
   - 기존 `02_data/` 노트와도 공통 컬럼이 있으면 연결한다.

6. **INDEX.md 갱신**
   - 루트의 `INDEX.md`를 읽는다.
   - 이번에 새로 만든 노트를 성격에 맞는 섹션(## 데이터, ## 검토 필요, ## 질문, ## 인사이트)에 링크로 추가한다.
   - 기존 내용은 지우지 않고 추가만 한다.

7. **작업 로그 기록**
   - `05_log/오늘날짜.md` (예: `05_log/2026-07-15.md`) 파일을 만들거나 있으면 이어서 작성한다.
   - 이번 실행에서 확인한 raw 파일 목록, 새로 만든 노트 목록, 검토 요청 목록, 연결한 위키링크를 간단히 기록한다.
   - `INDEX.md`의 `## 최근 로그` 섹션에도 이 로그 파일 링크를 추가한다.

작업 중 애매한 판단(표/자유형식 구분, 컬럼 의미, 위키링크 대상 등)이 계속 불확실하면 4번 규칙대로 검토 요청 메모로 남기고 넘어간다. 완료 후 사용자에게 새로 생성/수정된 파일 목록을 간단히 요약해서 보고한다.
